# Codex CLI 背景知識

這份文件用來補充技術分享前需要先理解的基本觀念。重點不是背名詞，而是知道這些概念會怎麼影響你使用 Codex CLI。

## Codex CLI 是 Agent，不只是聊天

Codex CLI 不是只能回答問題的聊天工具，而是可以在你的工作目錄裡協作的 agent。

它可以根據你的指令去讀檔、搜尋程式碼、執行命令、檢查 `git diff`，也可以直接修改檔案。也就是說，你不是只在問它「這段程式是什麼意思」，而是可以請它完成比較完整的工作，例如：

- 找出某個錯誤可能發生在哪些檔案。
- 根據現有文件補一段教學內容。
- 修改某個 function，並確認相關測試是否通過。
- 幫你 review 目前的修改範圍。

因此使用 Codex CLI 時，要把它當成「會動手做事的協作者」，而不是只會給建議的問答視窗。任務說得越清楚、範圍越明確，它越容易做出可檢查的結果。

## Codex 如何取得專案上下文

Codex 不會一開始就自動理解整個專案。它需要透過目前對話、你提供的需求、專案指引，以及實際讀取到的檔案內容來建立上下文。

常見的上下文來源包括：

- 你在對話中描述的任務目標和限制。
- 專案裡的 `AGENTS.md` 或其他規則文件。
- Codex 用 `rg` 找到的關鍵字、檔案路徑或引用位置。
- Codex 讀取的 Markdown、程式碼、設定檔或測試檔。
- `git status`、`git diff`、測試輸出、錯誤訊息等工具結果。

所以，給 Codex 任務時不需要一次貼上整個專案。比較好的方式是先說清楚目標，提供必要路徑或關鍵字，讓 Codex 自己去查相關檔案，再根據查到的內容執行下一步。

例如：

```powershell
請幫我看 background.md，補充 Codex CLI 初學者必須先知道的背景知識。
```

會比下面這種說法更容易開始：

```powershell
幫我整理一下這個專案。
```

因為前者有明確檔案和目標，Codex 可以直接建立相關上下文；後者範圍太大，通常需要再追問或先花很多時間探索。

## 為什麼提示中要使用 Markdown 格式

Codex 可以理解自然語言，但如果提示用 Markdown 寫得有結構，它會更容易分辨任務目標、限制、輸入資料和期望輸出。

對初學者來說，不需要寫很複雜的 prompt。先學會幾個 Markdown 格式就很有幫助：

- 用標題分段，例如 `## 目標`、`## 範圍`、`## 限制`、`## 驗收條件`。
- 用清單列出規則，避免把多個要求塞在同一句話裡。
- 用 code fence 包住命令、錯誤訊息、JSON 或程式碼。
- 用表格整理輸入與輸出欄位。
- 用核取清單列出完成條件。

例如：

````markdown
## 目標

請幫我補充 `background.md`，讓第一次使用 Codex CLI 的同事能理解必要背景知識。

## 範圍

- 只修改 `background.md`
- 使用繁體中文
- 不要重寫原本已經完成的段落

## 驗收條件

- 新增內容要適合初學者
- 每一節都要短而實用
- Markdown 格式不能壞掉
````

這種寫法比一整段口語描述更容易檢查，也更容易讓 Codex 照著你的期待工作。當任務變複雜時，Markdown 就像是把需求拆成清楚的小格子，讓模型和使用者都比較不容易漏掉重點。

## 使用 Codex 時的安全與驗證責任

Codex 可以幫你操作檔案和執行命令，但最後仍然需要由使用者負責確認結果。把 Codex 當成協作者，不代表可以跳過 review。

建議養成幾個基本習慣：

1. 開始前先看 `git status`，知道工作目錄目前是否乾淨。
2. 修改後用 `git diff` 檢查實際變更。
3. 文件修改要預覽 Markdown，確認標題、清單和 code fence 沒壞掉。
4. 程式碼修改要跑相關測試或至少執行可驗證的命令。
5. 不要把密碼、API key、cookie、token、客戶資料貼進對話或提交到 repo。

另外，Codex 對版本資訊、外部網站內容、CLI 最新行為不一定永遠正確。只要內容具有時效性，就應該用實際命令、官方文件或可靠來源再確認一次。

## 什麼是 Token？

`token` 是 LLM 模型可以處理和理解的最小文字單位，但它不等於「一個字」。

你可以先把 token 想成模型讀文字時切出來的小片段。英文裡，一個 token 可能是一個單字、半個單字、標點符號，或空白加上一段文字；中文裡，一個 token 可能接近一個字，也可能是多個字或符號組合，實際切法由模型使用的 tokenizer 決定。

簡單說：

- `token` 不是字數，也不是行數。
- 同一段內容，不同語言、符號、程式碼格式，token 數可能不同。
- 程式碼、JSON、Markdown 表格、長路徑、錯誤 log，通常會吃掉不少 token。
- 模型的輸入和輸出都會消耗 token。

可以用這個比喻對新手說明：

> 人類看文章時會用「字、詞、句子」理解內容；模型看內容時，會先把文字拆成 token，再根據 token 之間的關係理解意思。

所以在 Codex CLI 裡，如果你一次貼很多檔案、很多 log、很長的對話紀錄，模型不是「免費全部看完」，而是會占用 token 預算。

## 什麼是 Context Window？

`context window` 是模型在一次回應中可以同時參考的 token 空間上限。

它不是電腦記憶體，也不是模型永久記住你的所有對話。概念上更接近「模型這次回答前，桌面上最多可以攤開多少資料」。

你可以這樣解釋：

> context window 像一張有限大小的工作桌。你放得下需求、對話紀錄、檔案內容、錯誤訊息和工具輸出，模型就能同時參考它們；放太多時，就需要摘要、取捨，或把不重要的東西移出桌面。

在 Codex CLI 中，context window 通常會包含：

- 目前這輪你輸入的需求。
- 先前對話內容。
- Codex 讀到的檔案片段。
- 工具執行結果，例如 `rg`、`git diff`、測試輸出。
- `AGENTS.md` 或系統/專案指引。
- Codex 準備輸出的回答或修改計畫。

因此 context window 不是「記憶力有多好」，而是「這次工作時可同時放進模型視野的資料量」。

## Token 和 Context Window 的關係

token 是單位，context window 是容量。

可以用杯子比喻：

- token 是水滴。
- context window 是杯子的容量。
- 需求、檔案、log、對話紀錄都會往杯子裡倒水。
- 杯子快滿時，就要刪減、摘要、分段處理，或開新 session。

這也是為什麼長對話中會需要 `/compact`。它的目的不是讓模型變聰明，而是把前面的長對話壓縮成較短摘要，釋放 context window 空間，讓後續任務還能繼續放進重要資訊。

## 對 Codex CLI 使用者的實務影響

使用 Codex CLI 時，可以用幾個原則管理 token 和 context window：

1. 需求要清楚，但不用把整個專案一次貼上。
2. 大量錯誤 log 先貼關鍵段落，必要時再補完整輸出。
3. 請 Codex 先用 `rg`、`git diff`、目錄結構去找相關檔案，不要盲目讀全專案。
4. 任務變長時，使用 `/status` 觀察 session 狀態。
5. 對話太長但還要繼續同一件事時，使用 `/compact`。
6. 要開始完全不同的任務時，使用 `/new` 開新對話。
7. 專案規則放在 `AGENTS.md`，避免每次重複貼同樣規則。

## 分享時可以用的一句話版本

`token` 是模型閱讀文字的基本單位，不等於一個字；`context window` 是模型一次回答前能同時參考的 token 空間，不是永久記憶體。用 Codex CLI 時，輸入、檔案、log、對話紀錄都會占用這個空間，所以要學會精準描述、分段處理、適時 `/compact`。
